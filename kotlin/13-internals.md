## 13.1 How Kotlin Compiles to JVM Bytecode

Kotlin source code goes through several stages before becoming executable:

```
[.kt files]
    │
    ▼
[K2 Compiler Frontend]
    │ Parsing, semantic analysis, type inference
    ▼
[IR (Intermediate Representation)]
    │ Platform-agnostic, optimizable tree
    ▼
[JVM Backend]
    │ Generates JVM bytecode
    ▼
[.class files]
    │
    ▼
[JVM execution]
```

### Viewing Bytecode

IntelliJ IDEA lets you see the generated bytecode:
- `Tools → Kotlin → Show Kotlin Bytecode`
- Click `Decompile` to see equivalent Java code

This is one of the best ways to understand what Kotlin is really doing.

### A Simple Example

```kotlin
// Kotlin
fun add(a: Int, b: Int): Int = a + b
```

Decompiles to approximately:
```java
// Java equivalent
public static final int add(int a, int b) {
    return a + b;
}
```

Kotlin `Int` maps directly to Java `int` (primitive) — no boxing overhead for basic operations.

## 13.2 Data Classes Under the Hood

A Kotlin data class generates significant code:

```kotlin
// Kotlin source
data class Person(val name: String, val age: Int)
```

Decompiled Java equivalent:
```java
public final class Person {
    @NotNull
    private final String name;
    private final int age;

    public Person(@NotNull String name, int age) {
        this.name = name;
        this.age = age;
    }

    @NotNull
    public final String getName() { return this.name; }
    
    public final int getAge() { return this.age; }

    @NotNull
    public final Person copy(@NotNull String name, int age) {
        return new Person(name, age);
    }

    // Generated equals() — uses all primary constructor properties
    public boolean equals(@Nullable Object other) {
        if (this == other) return true;
        if (!(other instanceof Person)) return false;
        Person p = (Person) other;
        return Objects.equals(name, p.name) && age == p.age;
    }

    // Generated hashCode()
    public int hashCode() {
        return 31 * (name != null ? name.hashCode() : 0) + age;
    }

    // Generated toString()
    public String toString() {
        return "Person(name=" + name + ", age=" + age + ")";
    }

    // Destructuring component functions
    @NotNull
    public final String component1() { return this.name; }
    
    public final int component2() { return this.age; }
}
```

This explains why `data class` is so valuable — one line replaces 50+ lines of Java.

## 13.3 Lambdas in Bytecode

### Non-Inline Lambdas

A non-inline lambda creates an anonymous class (or uses invokedynamic in modern JVM):

```kotlin
val numbers = listOf(1, 2, 3)
numbers.forEach { println(it) }
```

Approximately compiled to (pre-JDK-8 style, simplified):
```java
numbers.forEach(new Function1<Integer, Unit>() {
    public Unit invoke(Integer it) {
        System.out.println(it);
        return Unit.INSTANCE;
    }
});
```

Or with `invokedynamic` (modern JVM optimization), it's more efficient and doesn't create a separate class file.

### Inline Functions — No Object Created

```kotlin
// Standard library forEach is inline:
inline fun <T> Iterable<T>.forEach(action: (T) -> Unit) {
    for (element in this) action(element)
}

numbers.forEach { println(it) }
```

Compiles to approximately:
```java
for (Integer element : numbers) {
    System.out.println(element);
}
```

No lambda object created — the lambda body is copied into the call site.

### Counting Lambda Allocations

```kotlin
// These are different:
fun countNonInline(list: List<Int>) {
    list.filter { it > 0 }  // creates a lambda object each call if not inlined
        .map { it * 2 }     // creates another lambda object
}

// Because filter and map are inline in the stdlib, actual bytecode is:
// for loop + conditional + another for loop — no allocations
```

## 13.4 Null Safety at the Bytecode Level

Kotlin's null safety translates to runtime checks in bytecode:

```kotlin
// Kotlin — null-safe
fun getLength(s: String): Int = s.length  // s is non-null
```

Compiled bytecode includes:
```java
// The @NotNull annotation generates a null check at the start:
public static final int getLength(@NotNull String s) {
    Intrinsics.checkNotNullParameter(s, "s");  // generated null check
    return s.length();
}
```

### Smart Casts in Bytecode

```kotlin
fun printIfString(obj: Any) {
    if (obj is String) {
        println(obj.length)  // smart cast — no explicit cast in Kotlin code
    }
}
```

Bytecode:
```java
public static final void printIfString(@NotNull Object obj) {
    if (obj instanceof String) {
        String s = (String) obj;  // explicit cast inserted by compiler
        System.out.println(s.length());
    }
}
```

### The !! Operator in Bytecode

```kotlin
val s: String? = getName()
println(s!!.length)
```

Bytecode:
```java
String s = getName();
if (s == null) {
    throw new NullPointerException("null cannot be cast to non-null type kotlin.String");
}
System.out.println(s.length());
```

This is exactly why `!!` is "explicit" about the NPE risk — it generates a checked throw.

## 13.5 When Classes Compile To

### Sealed Classes

```kotlin
sealed class Shape {
    class Circle(val radius: Double) : Shape()
    class Rectangle(val w: Double, val h: Double) : Shape()
}
```

Bytecode: `Shape` becomes an `abstract class`. `Shape.Circle` and `Shape.Rectangle` are inner classes that extend it. The `sealed` restriction is enforced by the Kotlin compiler — at bytecode level they're just regular final subclasses.

### Object Declarations (Singletons)

```kotlin
object Config {
    val timeout = 30
}
```

Compiles to:
```java
public final class Config {
    @NotNull
    public static final Config INSTANCE = new Config();  // singleton
    private final int timeout = 30;
    
    private Config() { }  // private constructor
    
    public final int getTimeout() { return this.timeout; }
}
```

### Extension Functions

```kotlin
// File: Utils.kt
fun String.shout() = uppercase() + "!!!"
```

Compiles to:
```java
// File: UtilsKt.class
public final class UtilsKt {
    @NotNull
    public static final String shout(@NotNull String $receiver) {
        return $receiver.toUpperCase() + "!!!";
    }
}
```

Extension functions are **static functions** that take the receiver as the first argument.

## 13.6 Generics and Type Erasure

Like Java, Kotlin generics are erased at runtime:

```kotlin
val list: List<String> = listOf("a", "b")
val list2: List<Int> = listOf(1, 2, 3)

// At runtime, both are just List — the type argument is gone
println(list is List<*>)    // true — can only check without type arg
// println(list is List<String>)  // ERROR — cannot check erased type at runtime
```

### Reified Types Bypass Erasure

```kotlin
inline fun <reified T> isListOf(list: List<*>): Boolean {
    return list.all { it is T }
}

// At runtime, T is preserved because of reified + inline
println(isListOf<String>(listOf("a", "b")))  // true
println(isListOf<Int>(listOf("a", "b")))     // false
```

Compiled — the type `T` is replaced with the actual type at the call site:
```java
// For isListOf<String>(...):
for (Object item : list) {
    if (!(item instanceof String)) return false;
}
return true;
```

## 13.7 Coroutines in Bytecode — A Peek

Coroutines are implemented as **state machines** in the bytecode. No OS-level primitives are used.

```kotlin
suspend fun fetchTwoItems(): Pair<String, String> {
    val a = fetchFirst()   // suspension point 1
    val b = fetchSecond()  // suspension point 2
    return Pair(a, b)
}
```

The Kotlin compiler converts this to a state machine class:

```java
// Approximately:
class FetchTwoItemsContinuation(private val completion: Continuation<Pair<String,String>>) 
    : ContinuationImpl {
    
    var state = 0
    var a: String? = null
    
    override fun invokeSuspend(result: Any?): Any? {
        return when (state) {
            0 -> {
                // State 0: start — call fetchFirst
                state = 1
                val r = fetchFirst(this)  // pass 'this' as continuation
                if (r == COROUTINE_SUSPENDED) return COROUTINE_SUSPENDED  // suspend!
                r  // if completed synchronously, fall through
            }
            1 -> {
                // State 1: fetchFirst completed — call fetchSecond
                a = result as String
                state = 2
                val r = fetchSecond(this)
                if (r == COROUTINE_SUSPENDED) return COROUTINE_SUSPENDED
                r
            }
            2 -> {
                // State 2: fetchSecond completed — return result
                val b = result as String
                Pair(a!!, b)
            }
            else -> throw IllegalStateException("...")
        }
    }
}
```

This is why coroutines have nearly zero overhead — they're just objects with state, not threads.

## 13.8 Performance Considerations

### Primitive Types vs Boxed Types

```kotlin
// Kotlin Int — compiled to Java int (primitive) when not nullable
val a: Int = 42        // int a = 42; — no boxing

// Kotlin Int? — compiled to Java Integer (boxed)
val b: Int? = 42       // Integer b = 42; — boxing occurs!

// In generics — always boxed (JVM limitation)
val list: List<Int> = listOf(1, 2, 3)  // List<Integer> — all boxed!

// For performance: use specialized primitive arrays
val intArray = IntArray(1000)        // int[] — no boxing
val array = Array<Int>(1000) { 0 }   // Array<Integer> — boxing!
```

### Avoiding Allocation in Hot Loops

```kotlin
// BAD — creates lambda objects in a hot loop (if not inline)
fun processHot(items: List<Int>) {
    for (i in 0..1_000_000) {
        items.forEach { /* lambda object created each time if not inline */ }
    }
}

// GOOD — stdlib functions are inline, so no objects created
fun processGood(items: List<Int>) {
    for (i in 0..1_000_000) {
        items.forEach { println(it) }  // forEach is inline — no allocation
    }
}
```

### String Concatenation

```kotlin
// BAD — creates intermediate String objects
val result = "Hello" + ", " + name + "! " + "You are " + age + " years old."

// GOOD — compiled to StringBuilder automatically
val result = "Hello, $name! You are $age years old."
// Or explicitly:
val result = buildString {
    append("Hello, ")
    append(name)
    append("! You are ")
    append(age)
    append(" years old.")
}
```

String templates ARE compiled to `StringBuilder` by the Kotlin compiler, so they're already optimized.

### Data Class Performance

```kotlin
data class Point(val x: Double, val y: Double)

// Heavy use with lots of copies — creates many objects
val p = Point(1.0, 2.0)
val moved = p.copy(x = p.x + 1.0)  // new object created

// For high-frequency use cases, consider a regular class with mutation:
class MutablePoint(var x: Double, var y: Double)
val mp = MutablePoint(1.0, 2.0)
mp.x += 1.0  // no allocation
```

### Value Classes (Inline Classes)

Kotlin 1.5+ introduced **value classes** — wrapper types that are inlined by the compiler:

```kotlin
@JvmInline
value class UserId(val value: String)

@JvmInline
value class Email(val value: String)

fun createUser(id: UserId, email: Email): User {
    return User(id.value, email.value)
}

val id = UserId("user-123")    // at runtime, just a String — no wrapper object!
val email = Email("a@b.com")   // also just a String at runtime

createUser(id, email)
// Compiled to: createUser("user-123", "a@b.com") — no boxing
```

Value classes give you type safety without runtime overhead.

## 13.9 The K2 Compiler

Kotlin 2.0 introduced the **K2 compiler** — a complete rewrite of the compiler frontend:

### What Changed

```
Old compiler:
[Source] → [PSI/AST] → [Frontend (type checking)] → [IR] → [Backend] → [Bytecode]
                          (old — slower, harder to maintain)

K2 compiler:
[Source] → [PSI/AST] → [K2 Frontend (type checking)] → [IR] → [Backend] → [Bytecode]
                         (new — 2-3x faster, better analysis)
```

### Performance Improvements

In real-world projects, K2 compilation is 2-3x faster:
- Better incremental compilation
- More efficient analysis algorithms
- Less redundant work

### Better Error Messages

K2 provides more precise, helpful error messages:
```
// Old compiler:
Error: Unresolved reference: getName

// K2 compiler:
Error: Unresolved reference 'getName'. Did you mean: 'name' (val property)?
```

## Complete Example: Understanding the Compilation

Let's trace how a complete Kotlin snippet compiles:

```kotlin
// Source: complex.kt
data class User(val name: String, val age: Int)

fun List<User>.adults(): List<User> = filter { it.age >= 18 }

fun processUsers(users: List<User>): String {
    val adults = users.adults()
    val names = adults.map { it.name }
    return names.joinToString(", ")
}
```

What the compiler generates:
1. `User` → Java class with `equals`, `hashCode`, `toString`, `copy`, `component1`, `component2`
2. `List<User>.adults()` → static method `ComplexKt.adults(List)` — takes `List` as first parameter
3. `filter { it.age >= 18 }` → because `filter` is `inline`, no lambda object created; instead, a for loop with an if check
4. `map { it.name }` → also `inline`; becomes a for loop
5. `joinToString` → calls `String.join` or similar, using `StringBuilder`
6. Return types are `@NotNull` annotated
7. Null checks generated for non-null parameter `users`

## Summary

Kotlin compiles to standard JVM bytecode, making it fully interoperable with Java. Data classes generate full boilerplate automatically. Extension functions are static methods. `object` declarations become singletons with a private constructor and a `INSTANCE` field. Null safety translates to runtime `Intrinsics.checkNotNullParameter` calls. Lambdas may create objects (non-inline) or be inlined at the call site (inline functions). Coroutines are state machines compiled by the Kotlin compiler — no OS threading needed. Type erasure is a JVM limitation; `reified` types bypass it for inline functions. The K2 compiler (Kotlin 2.0) brings 2-3x faster compilation.

---

## Key Takeaways

- View `Tools → Kotlin → Show Kotlin Bytecode` to understand what Kotlin generates
- `Int` is a JVM `int` (primitive) when non-nullable; `Int?` is `Integer` (boxed)
- Inline functions copy their body to the call site — zero lambda allocation
- Extension functions are static methods with the receiver as first parameter
- `object` declarations = private constructor + `INSTANCE` singleton field
- Coroutines = state machines (compiled by Kotlin) = lightweight, no OS threads
- Value classes (`@JvmInline value class`) are zero-cost wrappers
- String templates compile to `StringBuilder` internally — no concatenation overhead
- K2 compiler (default in Kotlin 2.0+) is 2-3x faster than the old frontend
